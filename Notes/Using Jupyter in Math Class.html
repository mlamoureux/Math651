Using Jupyter in Math Class¶

M. Lamoureux, March 11 2016¶

In the 2016 Winter term at University of Calgary, I started teaching a course on Industrial Mathematics (Math 651), with the goal of exposing students to some useful techniques for working on real industrial problems with a strong mathematical component.

One key step was to teach them to use a modern software tool called Jupyter, which allows one to combine text, graphical images, mathematical formulas, and functioning computer code all in one document. In fact, Jupyter allows one to create living documents where the users can play around with mathematics and document what they are doing, all in one single software package.

Jupyter works in the browser (Firefox, Chrome, Safari, etc) and uses a simple interface with the user. There is a choice of computational "kernels" that allow the user to write their code in standard languages such as R, Python, Julia, Matlab, and others. There is also a choice of infrastructure to host the system: beginners can use the site try.jupyter.org, more advanced users can use the PIMS hosting site jupyter.pims.math.ca or even install the system on their own computer (Windows, Mac OS, or Linux).

First impressions - instructor¶

I like the system a lot. In fact, I use it all the time now for creating lecture notes as well as computational examples. In Fall 2015 I created all my lecture notes for a complex analsyis course at UBC, which included a great deal of interesting numerical examples and complex mathematical graphics. Jupyter was great for this, in particular in that it seamlessly integrates English text, mathematical formulas, and computations with graphical output.

I especially like that I can easily write code in Matlab (which I have used for 20+ years), Python and Julia (which I am only just learning). By installing Jupyter on my own computer, I can also access audio devices on my computer, which is very useful for signal processing demonstations -- one of the things I do a lot in my work. (Note: this does not seem to work on PIMS Jupyter Hub, or on try.jupyter.org).

Some draw-backs.

  • To launch Juptyer on a Mac, you need to open a terminal and type in something on the command line. I have a hard time remembering typed commands, so eventually I wrote an AppleScript file that I just click on, which does the launch commands for me
  • I seem to have a hard time keeping track of my files. On try.jupyter.org, the files are somewhere out on the cloud and you have to upload and download your work. On jupyter.pims.math.ca, the files are on some server at UBC and it is not really obvious to me how to access them directly from my Mac. So I upload and download my work. On my own laptop's implementation of Jupyter, I usually can find the files, although Jupyter does not make the search easy.
  • Jupyter notebook, and the computational kernels, can be really slow at times. If you have a big file with a lot of mathematical formulas, Jupyter will be slow each time you open the file. It is like it is re-compiling all your TeX code every time you open the file.
  • the computational kernels (especially Julia) can be very slow at initializing. Say you want to use a new graphics package. You have to "Add" the package, which can mean waiting several minutes while the package is downloaded from the web, compiled, and installed. And God Forbid you should be disconnected from the Web. On my own laptop, eventually everything I use regulary gets installed and stored, so the waits go away. This is never the case on try.jupyter.ord (you get a fresh, un-packaged kernel each time), and on jupyter.pims.math.ca, the record is spotty -- sometime packages are saved and there for you next time, sometimes they are not.
  • these "delays" can be embarrasing when you teach a class, as you will try something easy (plot sin(x)), and have to wait a few minutes while the system installs somethign it needs.
  • there is currently no easy way to include a sketch in your notes. My work-around is to sketch on paper, take a picture with the camera on my laptop, and load that into the notebook. It works, but I would rather sketch with my mouse. (I think I will write code to do this!)

First impressions - students¶

The students first saw lecture notes from me in Jupyter, and were impressed that in a grad course, the prof would actually provide lecture notes, and that they were of good quality (with proper English, math formulas, images, and functioning computer code). They also got to see live demos on computations using Jupyter, which they really appreciated -- we saw numerical solutions to ODEs, simulations of stochastic processes for financial math, animations of chaotic behaviour in the Lorenz equations, Mandelbrot set, etc etc. I don't think they immediately realized that this was also a new system for me, because I was able to do a lot "right out of the box."

Second impressions - students¶

There was some disappointment when the students tried it out for themselves. The site try.jupyter.org is often slow and unreliable, so they quickly decided not to do anything serious on that site. We did a trial run of the PIMS-supported site jupyter.pims.math.ca and quickly crashed the system! With 10 students all trying to run graphics commands at the same time, some flaws in the PIMS setup immediately became apparent. (Ian Allison has tracked down these problems and tells me they are fixed now.) The students were also disappointed to discover that they could not run MATLAB on Jupyter (from PIMS) because of licensing restrictions.

As a result, most of the students decided to install Jupyter on their own computer, and link in a MATLAB or Octave kernel to do their computations.

Third impressions - students¶

Some students struggled on how to install Jupyter on their Windows machines. I use Mac OS, and I found it simple to install, so I don't understand why Windows was hard. A couple of students use Linux, and they told me it was trivial to install. We are sharing files on how to do things like install on Windows, link in various computational kernels, etc, so the students all now have access to a working copy of Jupyter notebook and with their favourite kernels. Of the ten students, we see they are using Matlab, Python, and R as their programming languages. About eight of the ten are using their own installation of Jupyter, while two are using the PIMS site to host their work.

Final thoughts¶

I am requiring the students to submit work on Jupyter notebooks, so they are actually being forced to use it. Their comfort zone is to use TeX for their formal reports, and MATLAB or R for their computations, and to cut-and-paste between the two if they need to add code or graphics. So this is a push on my part to get them to learn to use Jupyter effectively. In the long run, I think it is a better tool for communicating and collaborating on mathematics.

For the PIMS installation to be effective, we do need to resolve the licensing issue with MATLAB, so everyone can access this popular computational tool. We also need to resolve the speed issue when it comes to installing packages with the computational kernels. (What I would like to see is that each time I log in, the system remembers what packages I have installed in the past, and keeps them their available fo rme. So I don't have to wait while the system reloads and re-compiles my favourites). Finally, I think we need to resolve how we access files on the PIMS system. Ideally, I should be able to "mount" the PIMS file system onto my laptop, so I can see and access my files directly while using Jupyter on PIMS.

An example¶

One of my colleagues (Dr. Ryan Hamilton, at University of Calgary) had heard about Jupyter notebooks, and asked me for a demo. He is teaching a course on Functional Analysis, so I suggested doing a simple example to demonstrate an application of fixed point theory to the solution of ordinary differential equations. It took only a few minutes to write up the following note, which he will use in class.

Fixed point methods for solving ODEs¶

M. Lamoureux March 4, 2016¶

This method is called Picard's method. The idea is to start with a first order equation (or system of equations) in the form $$ y' = f(t,y(t)), \mbox{ with } y(t_0) = y_o$$ and convert to an integral form $$ y(t) = y_o + \int_{t_0}^t f(s,y(s)) \,ds. $$

The iteration runs by creating a sequence of functions $y_1(t), y_2(t), y_3(t), \ldots$ and hoping they converge to the answer. The first function in the sequence is $$ y_1(t) = y_o \mbox{ that is, it is the constant function with value } y_o.$$ Then the next function $y_2$ is given by integrating $f(t,y_1(t))$, so $$ y_2(t) = y_o + \int_{t_0}^t f(s,y_1(s)) \,ds, $$ and in gereral we will define $y_{k+1}$ in terms of the previous $y_k$ as $$y_{k+1}(t) = y_o + \int_{t_0}^t f(s,y_k(s)) \,ds. $$ With suitable conditions on the function $f(t,y)$, we can show this sequence converges to a solution of the ODE.

Let's do this in code. We use Julia here, because I am trying to learn this language, which is apparently good for math. You could easily do this in Python or Matlab.

Let's try to solve the simple ODE $$ y' = 2y \mbox{ with } y(0) = 1.$$ We know the exact answer is the exponential function $y(t) = \exp(2t)$.

For the numerical problem, we set up a time variable t (which is a vector of sample points from 0 to 1), a solutions variable y (which is an array, with one column for each approximate solution $y_k$, and a function formula $f = 2*y*dt$ which we will sum up in order to integrate.

In [65]:
t=linspace(0,1,100)    # the time variable, a vector of entries linearly spaced from 0 to 1.
dt = 1/100;            # step size for the integrals
y=zeros(100,10)        # an array for the solutions. Each column represents a function y_k

y[:,1] = 1             # this is our initial function guess, the constant function 1

for k = 1:9            
    f=2*y[:,k]*dt      # here we compute the integrand  2*y*dt
    f[1]=1             # this is y_o
    y[:,k+1] = cumsum(f,1)    # the cummulative sum is the Riemann sums of the integral
end

plot(t,y)              # now we plot the various functions that approximate the solution
ylim(.8,8)
Out[65]:

It is easy to see in the functions above that the $y_1$ function is the constant 1 function, the $y_2$ function is the linear function going straight from 1 to 3, the next one is a quadratic, and so on. These functions quickly converge to something that looks like an exponential.

Let try a harder ODE, like this: Let's try to solve the simple ODE $$ y' = (1-t)y \mbox{ with } y(0) = 1.$$ We expect something like exponential growth for $t$ close to zero, but once $(1-t)$ becomes negative, we should get rapid decay. Let's see if the numerics show this.

Okay, we see in the above plots that the solution is growing up to t=1, then decaying. Looks good.

But, you should try investigating longer intervals. This reveals that the behaviour can get weird for larger values of t. This suggests we have to be careful with Picard's method to be sure of convergence.

Here in the code and plot below, we see what happens on the t-interval $[0,5]$.

We see in the above that the approximate solutions are starting to blow up. Why?

It might be useful to solve the ODE exactly. It's pretty easy to find the exact solution is $$y(t) = \exp(t-t^2/2),$$ which should decay rapidly to zero.

Probably in the numerics, the approximate solution is going negative, which messes up everything.
